## Turn 1 — User (Discovery 에이전트 프롬프트)

```
당신은 MomTest 방식으로 도메인 규칙을 발견하는 Product Discovery / TDD 분석 에이전트입니다.

목표는 사용자의 요구사항이나 인터뷰 메모에서 테스트 가능한 계약을 추출하는 것입니다.
계약은 반드시 ID를 가져야 하며, 이 ID는 테스트와 구현을 연결하는 "추적의 못"으로 사용됩니다.

중요 원칙:
1. 사용자의 답을 유도하지 않습니다.
2. "이렇게 하면 되나요?" 식의 확인 질문을 하지 않습니다.
3. 과거 실제 사례, 실패 경험, 경계 상황, 예외 상황을 중심으로 묻습니다.
4. 추측으로 기능을 만들지 않습니다.
5. 근거가 없는 동작은 계약 ID로 만들지 않습니다.
6. ID에 없는 동작은 구현하지 않습니다.
7. AC, 비즈니스 규칙, 예외 규칙, 불변식을 구분합니다.
8. 최종 산출물은 테스트 가능한 계약표여야 합니다.

분석 대상 요구사항:
[여기에 요구사항, 인터뷰 메모, 사용자 설명을 붙여 넣으세요]

(Step 1~7 분석 절차 및 출력 형식 지시 — 생략 없이 원문 적용)
```

### 최근 실제 사례
1. 마지막으로 장바구니 금액이 틀렸다고 확인한 건 — 상품·수량·가격·기대/실제 금액?
2. "할인이 이상하다" CS 최근 사례 — 확인 화면·영수증·엑셀?
3. QA 중 직접 돌려본 입력·결과?

### 직접 계산·검증 경험
4. 손/계산기 검산 — 사용 공식·순서?
5. 엑셀 검산 — 열 구성·어긋난 지점?
6. 회계·정산 대조 — 소계/할인액/결제액 차이?

### 경계값·헷갈린 사례
7. 경계 금액(5만 원 등) 할인 적용 여부 사례?

### 잘못된 입력·장애
10. 음수 수량, 문자열 단가, 빈 칸 — 시스템/CS 반응?
11. 중복 행·같은 상품 두 번 — 합계 변화·문제 사례?
12. 입력 경로(API/폼/엑셀)별 결과 차이?

### 규칙 조합·적용 순서
13. 할인 2개 이상 — 순서로 금액 달라진 사례?
14. "같이 안 되는 할인" 위반 사례?
15. 할인 품목/비할인 품목 혼합 주문 사례?

### 규칙 위반 시 손실·오류
16. 금액 오류로 생긴 실제 손실?
17. "절대 나오면 안 되는 금액" 감사·약관 사례?

### 절대 허용 불가·수용 불가
18. "버그"로 분류된 금액·할인 관련 결과 2~3건?

### Boundary
21. 주문 폼/API 실제 입력 필드 나열?

## Turn 3 — User (인터뷰 메모 제공)

```
[인터뷰 메모]
- 2024-03-12: A 3개(12,000원) + B 1개(30,000원) = 66,000원. 5만 원 넘으면 10% 할인. 기대 59,400원.
- VIP 고객은 문턱 할인 후 추가 5% 할인. 순서는 문턱 → VIP 고정.

[실제 주문 사례]
- #1107: 소계 60,000 → 결제 54,000 (10% 할인)
- #1042: 소계 48,000 → 할인 없음

[엑셀 검산]
- 66,000 × 0.9 = 59,400 (ROUND)
- VIP 추가: 59,400 × 0.95 = 56,430

[CS 클레임 / 버그]
- 수량 -1 입력 시 화면 합계 0원 표시 (기대: 오류 또는 거부)
- items 없이 제출 시 500 에러

[명시적 비범위 / 미확인]
- 쿠폰·배송비: 언급 없음
- 50,000원 정확히: "넘으면"만 확인, 50,000 포함 여부 미확인
```

## Turn 4 — Assistant (Phase 1: 계약 추출)

**판정:** 제공된 근거만으로 계약 8건 추출. 미확인은 OOS.

### 추출된 계약 ID

| ID | 유형 | 한줄 요약 |
| -- | ---- | --------- |
| AC-1 | AC | 소계 > 50,000 → round(소계 × 0.9) |
| AC-2 | AC | 소계 < 50,000 → 할인 없음 |
| AC-3 | AC | VIP + 문턱 적용 시 round(round(소계×0.9)×0.95) |
| INV-1 | Invariant | 할인 순서: 문턱 → VIP |
| INV-2 | Invariant | 0 ≤ 결제금액 ≤ 소계 |
| INV-3 | Invariant | 소계 = Σ(price × qty) |
| E-1 | Error | qty < 0 → ValueError(인덱스), 0원 계산 금지 |
| E-2 | Error | items 없이 POST → HTTP 500 금지 |

### OOS (9건)

OOS-1 (50,000 경계), OOS-2 (VIP 단독), OOS-3~5 (쿠폰·배송·세금·포인트),
OOS-6~9 (미확인 Entity 경계·round 방식)

### 후속 MomTest 질문

- 소계 정확히 50,000원 주문 사례?
- 49,999 vs 50,001 1원 차이 CS 사례?

### 구현 우선순위

INV-3, E-1 → AC-2, AC-1 → AC-3, INV-1 → INV-2 → E-2
