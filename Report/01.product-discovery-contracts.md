## 1. 요약

인터뷰 메모·실제 주문·엑셀 검산·CS 클레임에서 **테스트 가능한 계약 8건**을 추출했다.
근거 없거나 미확인 항목은 **OOS 9건**으로 분류해 구현 범위에서 제외했다.

| 구분 | 건수 |
| ---- | ---- |
| Acceptance Criteria (AC) | 3 |
| Invariant (INV) | 3 |
| Error Contract (E) | 2 |
| Out of Scope (OOS) | 9 |

**구현·테스트는 반드시 계약 ID를 참조한다. ID 없는 동작은 만들지 않는다.**

---

## 2. 원본 근거 (인터뷰 메모)

```
[인터뷰 메모]
- 2024-03-12: A 3개(12,000원) + B 1개(30,000원) = 66,000원. 5만 원 넘으면 10% 할인. 기대 59,400원.
- VIP 고객은 문턱 할인 후 추가 5% 할인. 순서는 문턱 → VIP 고정.

[실제 주문 사례]
- #1107: 소계 60,000 → 결제 54,000 (10% 할인)
- #1042: 소계 48,000 → 할인 없음

[엑셀 검산]
- 66,000 × 0.9 = 59,400 (ROUND)
- VIP 추가: 59,400 × 0.95 = 56,430

[CS 클레임 / 버그]
- 수량 -1 입력 시 화면 합계 0원 표시 (기대: 오류 또는 거부)
- items 없이 제출 시 500 에러

[명시적 비범위 / 미확인]
- 쿠폰·배송비: 언급 없음
- 50,000원 정확히: "넘으면"만 확인, 50,000 포함 여부 미확인
```

---

## 3. 근거 레벨 분류

| 근거 | 레벨 | 분류 |
| ---- | ---- | ---- |
| 수량 -1 → 화면 0원 (기대: 오류/거부) | L0 | Error Contract |
| items 없이 제출 → 500 | L0 | Error Contract (Boundary) |
| 5만 원 **넘으면** 10% (#1107, #1042, 66,000 사례) | L1 | Acceptance Criteria |
| VIP: 문턱 할인 **후** 추가 5% (엑셀 59,400→56,430) | L1 + L2 | AC + 적용 순서 |
| 할인 순서: 문턱 → VIP 고정 | L2 | Invariant |
| 결제액 ≤ 소계 (모든 할인 사례) | L3 | Invariant |
| 50,000원 정확히 / VIP 단독 / 쿠폰·배송비 | — | Out of Scope |

---

## 4. 테스트 가능한 계약표

| ID | 계약(불변식 / 에러 / AC) | 근거 레벨 | 계층 | 테스트 아이디어 | 구현 범위 |
| -- | ----------------- | ----- | -- | -------- | ----- |
| **AC-1** | `소계 > 50_000`이면 `round(소계 × 0.9)`를 반환한다. | L1 | Entity | `#1107`: 60,000→54,000 / 인터뷰: 66,000→59,400 | `cart.py` |
| **AC-2** | `소계 < 50_000`이면 소계를 그대로 반환한다(할인 없음). | L1 | Entity | `#1042`: 48,000→48,000 | `cart.py` |
| **AC-3** | `is_vip=True`이고 AC-1이 적용되는 경우 `round(round(소계 × 0.9) × 0.95)`를 반환한다. | L1 | Entity | 엑셀: 66,000→59,400→56,430 | `cart.py` |
| **INV-1** | 할인 적용 순서는 **문턱 할인 → VIP 할인**이다. AC-3은 AC-1 결과에 VIP를 곱한다. | L2 | Entity | 66,000 VIP: 중간값 59,400 확인 후 최종 56,430 | `cart.py` |
| **INV-2** | 유효한 입력에 대해 `0 ≤ 결제금액 ≤ 소계`이다. | L3 | Entity | AC-1~3 케이스 + 무할인 케이스 property | `cart.py` |
| **INV-3** | `소계 = Σ(price[i] × qty[i])` (정수 원 단위). | L1 | Entity | A 3×12,000 + B 1×30,000 = 66,000 | `cart.py` |
| **E-1** | `qty[i] < 0`이면 인덱스 `i`를 포함한 `ValueError`를 발생시킨다. 0원으로 계산하지 않는다. | L0 | Entity | CS: qty=-1 → `ValueError`, not 0 | `cart.py` |
| **E-2** | `items` 필드 없이 POST하면 HTTP 500을 반환하지 않는다(검증 오류 응답). | L0 | Boundary | CS: 빈/누락 제출 → status ≠ 500 | `app.py` |

## 5. 계약별 테스트 문장

```
AC-1:  calculate(60000, is_vip=False) == 54000
       calculate(66000, is_vip=False) == 59400

AC-2:  calculate(48000, is_vip=False) == 48000

AC-3:  calculate(66000, is_vip=True) == 56430

INV-1: calculate(66000, is_vip=True) == round(59400 * 0.95)  # 문턱 후 VIP

INV-3: subtotal([{"price":12000,"qty":3},{"price":30000,"qty":1}]) == 66000

E-1:   calculate_items([{"price":1000,"qty":-1}]) raises ValueError matching "0"

E-2:   client.post("/order", data={}) → response.status_code != 500
```

## 6. 과잉 구현 방지 (OOS)

| OOS ID | 만들지 않을 동작 | 제외 이유 |
| ------ | --------- | ----- |
| **OOS-1** | `소계 == 50_000`일 때 할인 여부 | "넘으면"만 확인됨. 50,000 **포함** 여부 미확인 |
| **OOS-2** | VIP인데 소계 < 50_000일 때 VIP 5% 단독 적용 | "문턱 할인 **후**"만 언급. 해당 사례 없음 |
| **OOS-3** | 쿠폰·프로모션 코드 | 명시적 비범위 |
| **OOS-4** | 배송비 | 명시적 비범위 |
| **OOS-5** | 세금(VAT)·포인트·회원 등급(VIP boolean 외) | 근거 없음 |
| **OOS-6** | `price < 0`, `qty == 0`, 빈 장바구니 `[]`의 Entity 동작 | 사례·기대값 미제공 |
| **OOS-7** | `items is None` Entity `TypeError` | CS는 Boundary 제출 버그만 기술. Entity 계약 근거 없음 |
| **OOS-8** | 소수점 단가·통화 | 모든 사례가 정수 원 |
| **OOS-9** | `round` 대신 floor/ceil | 엑셀 `(ROUND)`만 확인. 0.5 경계 사례 없음 |
